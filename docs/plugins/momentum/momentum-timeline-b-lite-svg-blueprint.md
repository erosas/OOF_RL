document does not implement production UI, routes, persistence, match-data
writes, History integration, Session integration, Live integration, replay
sync, Overlay Lab integration, or Momentum Engine math changes.

Proposed path:

```text

```

This path is used because the repo already keeps Momentum Timeline planning and
validation docs under `docs/plugins/momentum`. A new
`docs/plugins/momentum-timeline` root would split related Timeline docs across
two locations without a current code ownership reason.

## 3. SVG Layer Architecture

Use one timeline SVG with stable class names and grouped layers:

```html
<svg class="mtl-root" viewBox="0 0 1200 180" role="img" aria-labelledby="mtl-title mtl-desc">
  <title id="mtl-title">Momentum Timeline</title>
  <desc id="mtl-desc">Fixture-only event-derived pressure timeline.</desc>
  <defs id="mtl-defs"></defs>
  <g class="mtl-background"></g>
  <g class="mtl-time-ticks"></g>
  <g class="mtl-momentum-bands"></g>
  <g class="mtl-segment-dividers"></g>
  <g class="mtl-selected-range"></g>
  <g class="mtl-event-markers"></g>
  <g class="mtl-marker-hit-targets"></g>
  <g class="mtl-focus-selection"></g>
  <g class="mtl-effects"></g>
</svg>
```

### `svg.mtl-root`

- Purpose: timeline viewport and SVG ownership boundary.
- Element types: root `svg`.
- Z-order: contains every SVG timeline layer.
- Data source: fixture metadata and derived render constants.
- Class naming: `mtl-root`.
- Accessibility: provide `title`, `desc`, and keyboard-accessible HTML controls
  or SVG focus targets for markers.
- Performance: one root SVG; avoid re-creating the whole DOM on every hover.

### `defs#mtl-defs`

- Purpose: gradients, small marker symbols, clip paths, filters.
- Element types: `linearGradient`, `radialGradient`, `symbol`, `clipPath`,
  limited `filter`.
- Z-order: non-rendering definitions.
- Data source: static tokens and icon strategy.
- Class naming: IDs prefixed `mtl-`.
- Accessibility: no direct interactive content.
- Performance: keep filters optional and never animate blur values in V1.

### `g.mtl-background`

- Purpose: rail backdrop and timeline panel base.
- Element types: `rect`, optional subtle grid lines.
- Z-order: bottom.
- Data source: static layout constants.
- Class naming: `mtl-rail-bg`, `mtl-panel-grid`.
- Accessibility: decorative; mark child shapes `aria-hidden="true"` if needed.
- Performance: static.

### `g.mtl-time-ticks`

- Purpose: major/minor time ticks and labels.
- Element types: `line`, `text`, optional small dots.
- Z-order: above background and below bands or visually aligned above rail.
- Data source: duration and overtime metadata.
- Class naming: `mtl-tick-major`, `mtl-tick-minor`, `mtl-time-label`.
- Accessibility: labels should be readable and not the only navigation method.
- Performance: static unless duration/zoom changes.

### `g.mtl-momentum-bands`

- Purpose: event-derived state segments along the rail.
- Element types: `rect` for V1; `path` only if tapered joins are needed.
- Z-order: rail body above background.
- Data source: `MomentumSegment[]`.
- Class naming: `mtl-band`, `mtl-band-blue-control`,
  `mtl-band-orange-control`, `mtl-band-contested`, `mtl-band-neutral`.
- Accessibility: include `aria-label` or related list text for selected segment.
- Performance: render one element per segment; avoid per-pixel gradients.

### `g.mtl-segment-dividers`

- Purpose: visible cuts between segments and OT boundary.
- Element types: `line`, narrow `rect`.
- Z-order: above bands, below selected range and markers.
- Data source: segment boundaries and `durationSeconds`.
- Class naming: `mtl-segment-divider`, `mtl-ot-divider`.
- Accessibility: decorative unless selected.
- Performance: static.

### `g.mtl-selected-range`

- Purpose: selected range bracket and badge connector.
- Element types: `rect`, `path`, `line`, `text`.
- Z-order: above bands/dividers, below markers if markers should sit on top.
- Data source: `SelectedMoment`.
- Class naming: `mtl-selection-box`, `mtl-selection-handle`,
  `mtl-selection-badge`.
- Accessibility: selected state must also be represented in the HTML details
  card and not only by color.
- Performance: update only this group on selection changes.

### `g.mtl-event-markers`

- Purpose: visible event markers.
- Element types: `g`, `circle`, `polygon`, `text`, optional `use` or `image`.
- Z-order: above rail and selected range.
- Data source: `EventMarker[]`.
- Class naming: `mtl-marker`, `mtl-marker-goal`, `mtl-marker-save`,
  `mtl-marker-epic-save`, `mtl-marker-assist`, `mtl-marker-shot`,
  `mtl-marker-demo`, `is-selected`, `is-low-confidence`.
- Accessibility: each marker should have a readable `aria-label` or matching
  HTML button.
- Performance: cap marker effects; use transforms for hover/focus.

### `g.mtl-marker-hit-targets`

- Purpose: larger invisible interaction targets for mouse/touch.
- Element types: transparent `circle` or `rect`.
- Z-order: above visible markers or parallel layer with pointer events.
- Data source: `EventMarker[]`.
- Class naming: `mtl-marker-hit-target`.
- Accessibility: prefer HTML button overlays for robust keyboard behavior, or
  make SVG hit targets focusable with labels.
- Performance: simple geometry only.

### `g.mtl-focus-selection`

- Purpose: focus ring and current keyboard selection affordance.
- Element types: `rect`, `circle`, `path`.
- Z-order: above markers.
- Data source: selected or focused marker/range.
- Class naming: `mtl-focus-ring`, `mtl-selected-marker-ring`.
- Accessibility: visible focus must not rely only on color.
- Performance: update only on focus/selection changes.

### Optional `g.mtl-effects`

- Purpose: restrained glow or pulses for selected state.
- Element types: low-opacity `circle`, `rect`, `path`.
- Z-order: usually between background and markers, or top for selection aura.
- Data source: selected state only.
- Class naming: `mtl-effect`, `mtl-effect-selected`.
- Accessibility: disable optional motion for reduced motion.
- Performance: no animation-heavy effects in V1.

## 4. Coordinate System

Recommended starting viewBox:

```text
viewBox="0 0 1200 180"
```

Constants:

```js
const padding = 40;
const railX = 56;
const railY = 58;
const railWidth = 1088;
const railHeight = 22;
const markerLaneY = 126;
const selectedBadgeY = 148;
const majorTickSpacing = 60; // seconds
const markerSize = 28;
const markerHitSize = 44;
```

Time model:

- Fixture stores elapsed seconds.
- `0` elapsed = match clock `5:00`.
- `300` elapsed = match clock `0:00`.
- Overtime extends beyond `300`.
- Display overtime as `OT +0:45`, `+0:45`, or equivalent depending on layout.

Derived duration:

```js
const regulationSeconds = fixture.durationSeconds; // normally 300
const totalSeconds = fixture.durationSeconds + fixture.overtimeSeconds;
```

Formula: second to x.

```js
function secondToX(second) {
  const clamped = Math.max(0, Math.min(second, totalSeconds));
  return railX + (clamped / totalSeconds) * railWidth;
}
```

Formula: x to second.

```js
function xToSecond(x) {
  const pct = Math.max(0, Math.min(1, (x - railX) / railWidth));
  return pct * totalSeconds;
}
```

Formula: match clock label.

```js
function formatMatchClock(second) {
  if (second <= 300) {
    const remaining = Math.max(0, 300 - Math.round(second));
    return Math.floor(remaining / 60) + ":" + String(remaining % 60).padStart(2, "0");
  }
  const overtime = Math.round(second - 300);
  return "+" + Math.floor(overtime / 60) + ":" + String(overtime % 60).padStart(2, "0");
}
```

Formula: selected range rectangle.

```js
function selectedRangeToRect(rangeStartSecond, rangeEndSecond) {
  const x1 = secondToX(rangeStartSecond);
  const x2 = secondToX(rangeEndSecond);
  return {
    x: Math.min(x1, x2),
    y: railY - 12,
    width: Math.max(2, Math.abs(x2 - x1)),
    height: railHeight + 42
  };
}
```

Marker lane assignment:

```js
function markerLane(events, event, index) {
  const near = events
    .slice(0, index)
    .filter((other) => Math.abs(secondToX(other.second) - secondToX(event.second)) < 32)
    .length;
  return near % 3; // lanes 0, 1, 2
}

function markerY(lane) {
  return markerLaneY + lane * 18;
}
```

Dense markers should preserve temporal x-position and use y-lane staggering,
not horizontal distortion, until a later clustering design is approved.

## 5. Timeline State Model

Fixture-only JSON shape:

```json
{
  "metadata": {
    "durationSeconds": 300,
    "overtimeSeconds": 45,
    "blueTeamName": "Zephyr FC",
    "orangeTeamName": "Nova United",
    "score": { "blue": 3, "orange": 2 },
    "fixtureLabel": "Fixture (Mock Data)"
  },
  "segments": [],
  "events": [],
  "selectedMoment": {},
  "contributions": []
}
```

Timeline metadata:

```ts
{
  durationSeconds: number,
  overtimeSeconds: number,
  blueTeamName: string,
  orangeTeamName: string,
  score: { blue: number, orange: number },
  fixtureLabel: string
}
```

Momentum segment:

```ts
{
  id: string,
  startSecond: number,
  endSecond: number,
  state: "blue_control" | "orange_control" | "contested" | "neutral",
  confidence: "high" | "medium" | "low",
  intensity: number,
  summary: string
}
```

Allowed states:

- `blue_control`
- `orange_control`
- `contested`
- `neutral`

Event marker:

```ts
{
  id: string,
  second: number,
  type: "goal" | "save" | "epic_save" | "assist" | "shot" | "demo",
  team: "blue" | "orange" | "neutral",
  playerName: string,
  segmentId: string,
  confidence: "high" | "medium" | "low",
  summary: string
}
```

Selected moment:

```ts
{
  selectionType: "segment" | "event" | "range",
  segmentId: string,
  eventId: string,
  rangeStartSecond: number,
  rangeEndSecond: number,
  badgeSecond: number
}
```

Contribution:

```ts
{
  playerName: string,
  team: "blue" | "orange",
  pressureContribution: number,
  momentumInfluence: number,
  contestInvolvement: number,
  source: "fixture" | "event_derived"
}
```

Safe labels for fixture copy:

- Blue Control
- Orange Control
- Contested
- Neutral / Reset
- Pressure Sequence
- Event-Derived Contribution
- Event-derived pressure/control signal

Avoid labels that imply possession, player positioning, rotation quality,
tactical advantage, coaching-grade certainty, or win odds.

Open asset decision:

- Inline SVG symbols provide styling and fewer network requests.
- External SVG files are easier to review as standalone assets.
- CSS mask icons are cheaper for dense markers but lose the richer mockup style.

### B. Current Fallback Path

Until icon assets are adopted:

- render marker hex/circle/reticle shapes directly in the timeline SVG,
- use short text labels,
- preserve event color identity,
- include readable `title`/ARIA text.

Fallback labels:

- goal: `GL`
- save: `SV`
- epic save: `ES`
- assist: `AS`
- shot: `SH`
- demo: `DM`

Marker size tiers:

- dense/default: 24px visual, 40px hit target,
- normal: 30px visual, 44px hit target,
- selected: 38px visual, 52px hit target,
- selected moment card icon: 72px to 96px.

Selected marker state:

- larger marker,
- double ring or bracket,
- visible focus ring,
- selected details card updates,
- label text remains visible in fallback mode.

Hover/focus state:

- do not rely only on glow or color,
- add outline/ring and marker lift,
- expose marker summary via `aria-label` or linked HTML control.

Dense marker collision handling:

- stagger markers across up to three vertical lanes,
- keep x-position faithful to time,
- if still dense, show stacked-count badge or focused expansion in later PR,
- never hide all markers of a supported event type.

Fallback behavior if icon is missing:

```js
function iconFor(type) {
  return iconManifest[type]?.available
    ? { mode: "asset", href: iconManifest[type].href }
    : { mode: "fallback", label: fallbackLabels[type] || "EV" };
}
```

## 8. Accessibility Plan

Keyboard selection:

- Event markers must be reachable by keyboard.
- Prefer HTML button overlays aligned to SVG marker coordinates for robust
  focus behavior.
- If using SVG focus targets, set `tabindex="0"` and `role="button"` with
  clear labels.
- Arrow keys may move previous/next marker in a later interaction PR.

Readable marker labels:

- Fallback labels must remain legible at marker size.
- Full event labels should appear in `aria-label`, tooltips, or the surrounding
  event list.

Selected state:

- Indicate selected state with shape/outline and details card, not only color.
- The selected timestamp badge should be text-readable.

ARIA strategy:

- SVG root gets `role="img"` with title/description for the overall timeline.
- Each marker gets an accessible name:
  `2:35 Epic Save by Zephyr FC, high confidence`.
- The selected moment card should mirror selected marker data in HTML.

Focus rings:

- Use a visible ring with high contrast.
- Do not hide focus outlines.

Color contrast:

- Check blue/orange/purple/gray bands against the dark rail.
- Text on colored badges should meet readable contrast.
